## Lecture on Model View Controller


Programming promjects launched by companies are usually built by team groups, for these websites and applications to be deployed online and get exploited by customers and partners, patterns and specific paradigms are required and need to be implemented.

The purpose of integrating set of rules and conventions, are also to preserve an organized codebase, that is readable, that can be easily debugged, and perfectly maintained by the the rest of the software engineer team.


The architetural paradigm that we are about to discuss is : Model View Controller 


The separation of concerns is applied on the frontend, the visible section by the user, and it can also be used in the backend, where the server and all the process going under the hood.
The MVC pattern which is located in the backend server, has that specific purpose. It enables the code to be split into diverse folders that has different missions.
That paradigm invented in 1978 by the Norwegian computer engineer Trygve Reenskaug, was specifically created to encapsulate and abstract codes into three different folders that intereact with each other:


·         Le Model qui représente la structure des données. Leur définition ainsi que les fonctions qui leur sont propres et qu’elles peuvent avoir. Ce module est complètement décoléré du code métier ou de l’affichage de telle sorte à ce que la modification de la logique ou de l’interface n’affecte pas la structure des données.
·         La View (ou les vues) représente l’interface graphique à livrer au client qui en fait la requête. Avoir le code lié à l’interface isolé de la logique métier ou des données permet de faire des modifications à l’interface graphique sans avoir à se soucier de casser du code métier ou la structure des données.
·         Le(s) Controller(s) sont au cœur de la logique métier de votre application. Ils se situent entre les vues et le model. Les requêtes qu’un client va faire depuis l’interface graphique, la view, vont être dirigées vers un controller qui sera en charge de manipuler les données dont il a besoin avec la brique Model, la traiter suivant le besoin métier, puis ordonner à la view de répondre au client avec les bons éléments.
-Model : the model contains all the information, data, pictures stored into relational database structures such as SQL or non-relational object oriented databases such as Mongodb.

-Views: will show the data retrieved by Model into the clientside, that is the visible part of the pattern. Views is directly connected with the controllers.

-Controllers : are the middleman, the most important piece of the programming process. The browser sending a request to the server will be received by different backend layers :

         •The Router: the router functionality is to listen to the URL client-side request, so it can identify which CRUD req is sent, so it can redirect it to the adequate controller

         •The Controller: which is also a MVC backend programming component, receives the req from the router, and matches it with the adequate router, that will execute the requested action sent by the browser, create/poste, read/get, update/put, delete.
      Note: According to the nature of the req, the controller can either directly execute an action to view (ex: refresh a page), either go through models to collect/retrieve required data.

These three patterns help organize and visualize how the project codebase works underneath the hood, to have a better understanding of each functionalities and their missions. The Model is a sort of bucket connecting the server with a database that store all information data needed for further requests, mongodb is my favorite non-relational database. 
Its specificity is that, it assemble data into collections of document, that are array of objects prototype. The models folder regroup these specific files that will be rendered to front end templates (ejs, react, hug), so views can shw them in the front end. Controller connect the right req with the right controller so the actions requested can be executed and the browser can receive the  according response.
